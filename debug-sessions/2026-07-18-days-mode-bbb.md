# 2026-07-18 days 模式改用柏宝书 API 判断日期推进

## 起因
用户反馈：设的是 `linesMode='days'`（按游戏内日期推进事件线），实际几乎每楼都在触发 `shouldAdvance=true` → `runGenerateLines` → 烧 token。

## 根因
`detectInGameDayChange`（旧 index.js:247）原本靠 5 条正则扫 AI 消息文本猜"日期变了没"：

```js
const DAY_PATTERNS = [
    /第\s*(\d+)\s*[天日]/,
    /([一二三四五六七八九十百千\d]+)\s*月\s*([一二三四五六七八九十百千\d]+)\s*[日号]/,
    /星期([一二三四五六日天])|周([一二三四五六日天])/,
    /今天|明天|后天|大后天/,        // ← 每条 RP 都命中
    /day\s*(\d+)/i,
];
```

首匹配 `m[0]` 存进 `_lastDetectedDay`，两条消息只要命中的字符串不同就算"日期变了"。三个坑叠一起：

1. **相对词滥触发**：`今天/明天/后天/大后天` 每条对话都会出现
2. **弱信号 `星期X|周X`** 被角色顺口一提就中招
3. **同一天异写法误判**：`第3天` vs `第3日`、`星期一` vs `周一`、`3月5日` vs `2024年3月5日` —— 都是同一天，字符串不等 → 误触发

## 方案：改读柏宝书 state.time
柏宝书（ST-BaiBai-Book）已经维护了权威的游戏内时间。改成读它的公开 API：

- 全局对象 `globalThis.STBaiBaiBook.getSnapshot({ floor, at: 'after' })` → `{ state: { time: '...' } }`
- 就绪事件 `st-baibai-book:ready`（本次没订阅，运行时检查全局对象足够）

三种实测 `state.time` 格式（从用户现有聊天的 `extra.bbs_leaf.delta` 里抓的）：
- `1287/04/01 08:55`
- `2024-10-07 18:35`
- `大陆历1686年10月29日 20:30`

抽日 key 用正则短路匹配：
```
YYYY年M月D日 / YYYY[/-.]M[/-.]D  → "1686-10-29"
第N天/day N                       → "day-3"
古代中文（元和三年腊月初八）           → "cn-3-12-8"
无法识别                          → null（不推进）
```

关键细节：
- 用 `${+m[1]}` 剥前导零：`04` ≡ `4`，`YYYY/04/01` ≡ `YYYY年4月1日`
- era 前缀（"大陆历"、"神纪"）被 `\d{2,4}` 前的部分吞掉，不参与 key
- 时分秒尾巴不参与匹配，同天不同时点 key 相等
- 中文数字支持 0–99（`一二三四五六七八九十百` 组合），加 `正/冬/腊` 月别名、`初X` 日别名、`元年 = 1 年`
- 阿拉伯 vs 中文两个 key 命名空间分开（`1686-10-29` vs `cn-3-12-8`）避免碰撞

柏宝书不可用时（`globalThis.STBaiBaiBook` 缺失）：`console.info` 一次性提示 + 返回 false，宁可不推进也不误报。

## 单测（node inline）
```
'大陆历1686年10月29日 20:30'  → 1686-10-29
'1287/04/01 08:55'          → 1287-4-1
'2024-10-07 18:35'          → 2024-10-7
'元和三年腊月初八'             → cn-3-12-8
'贞观十四年正月十五'            → cn-14-1-15
'傍晚' / '' / null           → null（安全）
1287/04/01 == 1287/4/1     → true（padding 无关）
1686年10月29日 22:00 == 1686年10月30日 00:05 → false（跨天）
元和三年腊月三十 == 元和四年正月初一        → false（跨年）
```

## 需要复盘的坑
- **BBB 就绪时序**：假设 CHARACTER_MESSAGE_RENDERED 时 BBB 已经处理完 leaf（因为 BBB 是 generate_interceptor，跑在生成期间，早于渲染事件）。如果实际有竞态，某楼首次触发时 snapshot 可能没包含最新一楼的 delta → `at: 'after'` 会漏。此时下一楼会接住，最多迟一楼推进。
- **纯白天/夜晚跨天**：如果柏宝书的 `state.time` 只写"傍晚"→"深夜"不带日期，抽日返回 null，不推进。这是柏宝书语义问题，不是本层能兜的。
- **excludeCurrent 语义搬迁**：旧实现读的是"上一条 AI 消息的文本"，新实现读的是"上一条 AI 楼的 BBB 快照 at:after"——语义相近但不完全相同。观察一下有没有意外行为。
